dbos-clj · open-source library

Extracting a reusable DBOS wrapper for Clojure

Kapital and the Digital Worker PoC ship two nearly identical hand-written wrappers around dbos-transact-java 1.0.0. This plan extracts the shared plumbing — serializer, step macros, instance lifecycle, client, query — into a stateless, backend-agnostic library (dbos-clj), with logging routed through the Trove facade so each service keeps its own backend (μ/log for Kapital, Telemere for Digital Worker).

1Overview

DBOS Transact 1.0.0 is instance-based: you build a DBOS from a DBOSConfig, register queues and workflows through the (public-but-internal) DBOSIntegration, launch, and shutdown. Workflows are plain Clojure fns of [dbos input] — no Java interfaces, deftypes or proxies. Both Kapital (ohana.core.dbos.* + ohana.kapital.server.dbos.core) and the Digital Worker PoC (saas.dbos.*) independently arrived at the same shape around this API, differing only in (a) how the instance is mounted into the system and (b) which logging library the helpers call.

The goal is a small, importable library that owns the DBOS interop and nothing else: no defstate, no Integrant keys, no app config. It exposes pure functions to build/launch/shutdown an instance and register workflows; the services keep ownership of lifecycle (Mount / Integrant), datasource, queue names, and workflow definitions.

3Design

Following the sketch in new-dbos-lib.org (utils → core since the lib is stateless), under the existing top-level dbos root:

Interop stays confined. All dev.dbos.transact.internal.DBOSIntegration usage lives in dbos.core (registration, start-workflow!, start-child-workflow!) — exactly as both repos already isolate it. Nothing outside the library touches internal DBOS classes.

3.2 Stateless instance lifecycle (also in dbos.core)

In both source repos, instance setup lived in its own namespace only because that namespace held the Mount/Integrant state — the split tracked the instantiation boundary, not a concern boundary. This library never instantiates: create/launch!/shutdown! are plain functions the consumer wires into its own state cell. With no state to isolate, there's no reason to keep a separate dbos.instance — the config builder, registration, and lifecycle fns sit in dbos.core alongside the workflow helpers. create builds and registers but does not launch, so the caller controls when the executor goes live:

(ns dbos.core
  "Stateless DBOS surface: workflow-body helpers + step macros AND instance
  setup/registration. Instantiation and lifecycle ownership belong to the
  consumer (Mount/Integrant/etc.)."
  (:require [dbos.serializer :as serializer]
            [taoensso.trove :as trove])
  (:import (dev.dbos.transact DBOS)
           (dev.dbos.transact.config DBOSConfig)
           (dev.dbos.transact.workflow Queue)))

(defn dbos-config
  "Build a DBOSConfig from a plain map. Only :datasource and :app-name are
  required; everything else is optional and generalizes both consumers."
  ^DBOSConfig
  [{:keys [datasource app-name app-version executor-id serializer
           schema listen-queues admin-server? conductor]}]
  (cond-> (-> (DBOSConfig/defaults app-name)
              (.withDataSource datasource)
              (.withSerializer (or serializer (serializer/transit-serializer))))
    executor-id      (.withExecutorId executor-id)
    app-version      (.withAppVersion app-version)
    schema           (.withDatabaseSchema schema)
    (seq listen-queues) (.withListenQueues (into-array String listen-queues))
    admin-server?    (.withAdminServer true)
    (:domain conductor)  (.withConductorDomain (:domain conductor))
    (:api-key conductor) (.withConductorKey (:api-key conductor))))

(defn create
  "Construct + register queues + register workflows. Does NOT launch.
  Returns {:dbos/instance DBOS :dbos/workflow-registry {kw -> definition}}."
  [{:keys [config queues workflows]}]
  (let [instance (DBOS. (dbos-config config))]
    (doseq [^Queue q queues] (.registerQueue instance q))
    {:dbos/instance instance
     :dbos/workflow-registry (register-workflows! instance workflows)}))

(defn launch!   [{:dbos/keys [^DBOS instance]}] (.launch instance))
(defn shutdown! [{:dbos/keys [^DBOS instance]}] (.shutdown instance))

Kept generic, not opinionated

Executor-id resolution, worker-vs-app listen-queues policy, and the conductor on/off toggle were Kapital-specific decisions. The lib takes them as plain inputs; the consumer computes them. Digital Worker simply omits listen-queues and conductor.

Scheduling stays in the lib

Scheduled workflows need the exact (Instant, Object) method signature via the ScheduledWorkflowFn bridge, plus apply-schedules! after launch. This is fiddly interop worth centralizing — exposed as an optional :workflow/schedule key on the definition, a no-op when absent (Digital Worker never uses it).

3.3 Workflow definition validation (pure Clojure, no Malli)

Both source repos validate the definition map with Malli (mc/validate + me/humanize). A general-purpose library shouldn't drag in Malli for a four-key check — the shape is small and fixed, so a hand-written predicate collecting a vector of human-readable errors is enough and keeps the dependency footprint minimal. The rules are the Kapital superset (adds the optional :workflow/schedule). A workflow fn is [dbos input]; deps are closed over at registration by the consumer (via partial, as Digital Worker's ig/init-key already does):

(defn workflow-definition-errors
  "Return a vector of human-readable problems with a workflow definition,
  or nil when it's valid. Pure Clojure — no schema library."
  [{wf-name :workflow/name
    wf-fn :workflow/fn
    max-attempts :workflow/max-recovery-attempts
    schedule :workflow/schedule
    :as definition}]
  (not-empty
   (cond-> []
     (not (map? definition))
     (conj "definition must be a map")

     (not (and (keyword? wf-name) (namespace wf-name)))
     (conj ":workflow/name must be a namespaced keyword")

     (not (fn? wf-fn))
     (conj ":workflow/fn must be a function of [dbos input]")

     (and (some? max-attempts) (not (int? max-attempts)))
     (conj ":workflow/max-recovery-attempts must be an integer")

     (and (some? schedule)
          (not (and (map? schedule)
                    (string? (:cron schedule))
                    (string? (:queue schedule)))))
     (conj ":workflow/schedule must be {:cron string :queue string}"))))

(defn validate-workflow!
  "Throws ex-info with the collected errors when `definition` is invalid;
  returns it unchanged when valid."
  [definition]
  (when-let [errors (workflow-definition-errors definition)]
    (trove/log! {:level :error :id :dbos.workflow/invalid-definition
                 :data {:workflow/name (:workflow/name definition)
                        :errors errors}})
    (throw (ex-info "Invalid workflow definition"
                    {:workflow/name (:workflow/name definition)
                     :workflow/validation-errors errors})))
  definition)

Same contract, fewer deps. validate-workflow! keeps the exact behavior both repos rely on — log the problem under a structured id, then throw an ex-info carrying :workflow/validation-errors — so the failure surface at registration is unchanged. Only the mechanism (predicate vs Malli) differs.

3.4 Serializer with caller-injected time handlers

The serializer is byte-for-byte shareable except the transit.time require — which is exactly the piece we don't bundle. The luminus java.time handlers are app-owned data (both services already keep a *.common.transit.time copy for their HTTP/websocket wire format); vendoring them into the library would fork that copy and quietly drift. Instead the library owns only the DBOS-specific machinery (the java-object Jackson box, throwable mapping, the frozen format id) and takes transit handlers as an argument:

(defn transit-serializer
  "DBOSSerializer backed by Transit json-verbose. The caller supplies the
  transit :write-handlers / :read-handlers (e.g. their app-wide java.time
  handlers). Anything with no handler falls through to the java-object box."
  (^DBOSSerializer [] (transit-serializer nil))
  (^DBOSSerializer [{:keys [write-handlers read-handlers]}]
   (reify DBOSSerializer
     (name [_] serializer-name)            ; "transit_json_verbose" — FROZEN
     (serialize   [_ v] (serialize v write-handlers))
     (deserialize [_ t] (deserialize t read-handlers))
     (serializeThrowable   [_ th] (serialize (throwable->data th) write-handlers))
     (deserializeThrowable [_ t]  (some-> t (deserialize read-handlers) data->throwable)))))

No handlers is a valid default. Called with no opts, the serializer still works: Clojure data (keywords, UUIDs, collections) round-trips natively, and any java.time value simply falls through to the java-object Jackson box rather than getting a compact tag. Kapital and Digital Worker pass their existing *.common.transit.time handlers to get first-class java.time fidelity and human-readable rows — same as today.

Frozen format id. serializer-name = "transit_json_verbose" is recorded in every DBOS row. It must never change once workflow data exists in either service's system DB — so the constant is copied verbatim, and the java-object Jackson box behavior (fail-fast on unserializable, reconstruct on read) is preserved exactly.

3.5 Client & query become plain functions

Kapital's client is a Mount defstate that constructs a DBOSClient with the transit serializer, plus enqueue-workflow!. In the stateless lib, construction is a function and the client is an argument — the consumer owns the state cell and the queue name:

(defn create-client
  "Build a DBOSClient wired with the transit serializer. Caller owns lifecycle
  (.close on shutdown)."
  ^DBOSClient
  ([datasource] (create-client datasource nil (serializer/transit-serializer)))
  ([datasource schema serializer] (DBOSClient. datasource schema serializer)))

(defn enqueue-workflow!
  "Enqueue by workflow keyword onto `queue`. (workflowName, className) derived
  via workflow-identity — no client-side registry, typos surface as durable
  NOT_FOUND at the worker."
  [^DBOSClient client queue wf-key workflow-id workflow-data]
  (let [{:keys [wf-name class-name]} (dbos/workflow-identity wf-key)
        opts (dbos/->enqueue-options {:workflow-name wf-name :class-name class-name
                                      :queue queue :workflow-id workflow-id})]
    (dbos/add-derefable
     (.enqueueWorkflow client opts
                       (object-array [(assoc workflow-data :workflow/id workflow-id)])))))

No lifecycle helper for the client — only create-client. The core DBOS instance gets create/launch!/shutdown! because shutdown! has real work to do: it must drain workflows still in flight before the executor goes away. A DBOSClient is just a stateless enqueuer — it holds no running workflows and has no such obligation — so wrapping it in a launch/shutdown lifecycle would be ceremony with no payload. The library exposes create-client and stops there; the consumer .closes it whenever (or never) it likes.

3.6 Query is protocol-based over both DBOS and DBOSClient

Kapital's query today takes a DBOSClient only. But the same two methods — getWorkflowStatus(String) and listWorkflows(ListWorkflowsInput) — exist with identical signatures on both dev.dbos.transact.DBOS (the in-process instance) and dev.dbos.transact.DBOSClient (the out-of-process enqueuer). So the library exposes a protocol and extends it onto both classes, letting a caller query whichever handle they hold — the running instance inside the worker, or a client from an app node:

(ns dbos.query
  (:import (dev.dbos.transact DBOS DBOSClient)
           (dev.dbos.transact.workflow ListWorkflowsInput WorkflowStatus)))

(defprotocol WorkflowQueryable
  "Read-side of DBOS, satisfied by both a DBOS instance and a DBOSClient."
  (-list-workflows      [this ^ListWorkflowsInput input])
  (-get-workflow-status [this ^String workflow-id]))

(extend-protocol WorkflowQueryable
  DBOS
  (-list-workflows      [this input]       (.listWorkflows this input))
  (-get-workflow-status [this workflow-id] (.getWorkflowStatus this workflow-id))
  DBOSClient
  (-list-workflows      [this input]       (.listWorkflows this input))
  (-get-workflow-status [this workflow-id] (.getWorkflowStatus this workflow-id)))

;; Public fns take Clojure maps in, Clojure maps out — dispatch is on the
;; queryable's type, so the SAME call works on either handle:
(defn list-workflows
  "(list-workflows instance query-params) or (list-workflows client query-params)."
  [queryable query-params]
  (->> (->list-workflows-input query-params)
       (-list-workflows queryable)
       (mapv workflow-status->map)))

(defn get-workflow-status
  [queryable workflow-id]
  (some-> (-get-workflow-status queryable workflow-id)
          (.orElse nil)
          workflow-status->map))

The ->list-workflows-input and workflow-status->map converters move in unchanged; only the entry points become protocol-dispatched. dbos.constants is pure data and moves in unchanged.

Why a protocol, not two arities or a type check. The two classes share no common Java interface for these reads, so dispatch has to key on concrete type. A protocol keeps the public API a single (list-workflows queryable input) shape, is open for extension (a test double or a future queryable can extend it), and avoids an instance? ladder in the hot path.

Option A — split core vs instance

Mirrors the two source repos. Rejected here: their split existed only because instance carried the Mount/Integrant state. This lib is stateless, so the split would draw a line where there's no longer a concern boundary — just extra indirection.

Option B — dbos.core holds the instance surface (chosen)

Workflow-body helpers and instance setup/registration are the same "you have a DBOS instance" surface; they belong together. client/query/serializer/ constants stay separate — they are genuinely independent concerns (out-of-process interaction, wire format, data), not lifecycle.

3.7 Considered & rejected (for now): defworkflow sugar + registry NOT SHIPPING

Decision: not shipping. The explicit definitions-vector path stays the only registration API — it works for both Mount and DI consumers and keeps dbos.core free of a load-time global. The analysis below is kept as a record of why, in case a future consumer makes the case for revisiting it.

Today a workflow is a plain def of a definition map, and each namespace exposes a definitions vector the consumer into-s into one big workflow-registrations. It's explicit and data-first, but it has boilerplate: you write the map, then remember to add the namespace's definitions to the aggregate. The proposed sugar collapses "define the fn" and "register it" into one form: 

(defworkflow :app.workflow/cool-workflow
  [dbos input]
  ;; …continue as a normal function body…
  )

;; corrected expansion (the sketch's @body / #function-name won't compile):
(defmacro defworkflow
  "Define a workflow fn AND register its definition. Optional opts map may
  carry :workflow/schedule and :workflow/max-recovery-attempts."
  [wf-key & body]
  (let [[opts params & fn-body] (if (map? (first body))
                                 body
                                 (cons nil body))
        sym (symbol (name wf-key))]      ; :a.b/cool-workflow -> cool-workflow
    `(let [v# (defn ~sym ~params ~@fn-body)]
       (registry/register!
        (merge {:workflow/name ~wf-key :workflow/fn ~sym} ~opts))
       v#)))

The value is real, but so are the rough edges — this is why it stays optional and off the critical path:

Rough edge — reintroduces global state

registry/register! implies a mutable global (an atom of kw → definition). A stateless library growing a load-time global is exactly the coupling this extraction set out to remove. It's "def-time data" like a multimethod table, not a live DBOS instance — but it's still ambient state the consumer no longer controls.

Rough edge — implicit load order

The registry is only complete once every workflow namespace is required for its side effects. That's the classic "required only to populate a global" smell. The current definitions-vector approach makes the dependency explicit and greppable; Digital Worker even uses an #ig/refset specifically to avoid load-order magic.

Rough edge — clashes with dependency injection

A bare [dbos input] fn registered at load time can't close over runtime deps. Kapital's workflows pull deps from global Mount states, so this fits; Digital Worker injects deps ((partial dummy-workflow deps) in ig/init-key) and has no seam here. defworkflow quietly assumes the Mount/ambient-deps model — it would not serve the DI consumer.

Rough edge — where do opts go?

Definitions carry :workflow/schedule and :workflow/max-recovery-attempts. The terse (defworkflow kw params & body) has nowhere to put them without an opts map arg, which erodes the "it's just a defn" simplicity that made the sugar attractive.

Leaning: offer it as opt-in sugar over the same data, not a replacement. Keep the explicit definitions-vector path as the primary API (works for both Mount and DI consumers). If defworkflow ships, make it a thin, separate namespace (e.g. dbos.registry) whose global is clearly opt-in and documented, and whose macro expands to the exact same definition map the vector path uses — so a consumer can mix both, and the DI consumer can simply ignore it. That keeps the core stateless and lets the sugar be a convenience, not a fork in the model. Not committing until we're convinced it earns its keep.

4Logging via Trove

Trove is a minimal facade (1 macro, 0 deps) for library authors: emit rich, structured logs without forcing a backend on consumers. Its entire public surface is three things — the log! macro, the *log-fn* dynamic var, and set-log-fn! — using the same map API as Telemere ({:level :id :data :msg}). The backend log-fn receives [ns coords level id lazy_] where lazy_ forces to {:keys [msg data error kvs]}. Trove ships ready-made backend fns for both loggers already in play: taoensso.trove.mulog and taoensso.trove.telemere, chosen by setting *log-fn*. 

;; --- in kapital's entrypoint: choose μ/log ---
(trove/set-log-fn! (taoensso.trove.mulog/get-log-fn))

;; --- in digital-worker's entrypoint: choose Telemere ---
(trove/set-log-fn! (taoensso.trove.telemere/get-log-fn))

4.1 Contextual logging: Trove has no facade — inject one

Trove core deliberately has no with-context equivalent; each log! carries only its own :data. Context isn't lost, though: because Trove just delegates to the backend, the backend's native context still enriches Trove logs — the mulog backend calls ml/log*, so μ/log's u/with-context merges into every routed event; Telemere's t/with-ctx+ behaves the same. The snag is that those macros are backend-specific, and the library can't call one without picking a backend.

So the library injects the context hook the same way it injects the log fn: a dynamic *context-fn* of [ctx-map thunk], defaulting to a no-op, set once at startup to the backend's context mechanism. The step macros wrap their body in it, restoring the exact behavior Kapital's macros had (the :workflow/step tag riding along on every nested log) — now backend-agnostic: 

;; --- inside the library ---
(ns dbos.core (:require [taoensso.trove :as trove]))

(def ^:dynamic *context-fn*
  "Pluggable contextual-logging hook: a fn of [ctx-map thunk] that runs `thunk`
  with `ctx-map` merged into the backend's ambient log context, so every log
  emitted inside (including nested workflow-body logs) carries it. No-op by
  default. Set once at startup next to trove/*log-fn*."
  (fn [_ctx thunk] (thunk)))

(defmacro run-step
  "Executes a workflow step that RETURNS a value (persisted for recovery)."
  [dbos step-name & body]
  `(let [step-name# ~step-name]
     (*context-fn* {:workflow/step step-name#}
       (fn []
         (trove/log! {:level :debug :id :dbos.workflow/step-start
                      :data {:workflow/step step-name#}})
         (execute-step ~dbos step-name# (fn [] ~@body))))))

;; --- consumer wiring: one line, alongside set-log-fn! ---
;; kapital (μ/log):
(alter-var-root #'dbos.core/*context-fn*
  (constantly (fn [ctx f] (u/with-context ctx (f)))))
;; digital-worker (Telemere):
(alter-var-root #'dbos.core/*context-fn*
  (constantly (fn [ctx f] (t/with-ctx+ ctx (f)))))

Both mechanisms are fn-wrappable. u/with-context and t/with-ctx+ are macros, but each wraps trivially in a (fn [ctx f] (… ctx (f))) — so a single injected fn covers either backend, and consumers that don't care get the no-op default (step name still lands in :data).

This directly delivers the ask: the two convenient step macros (run-step/do-step!) move into the library with their logging and contextual tagging intact, and each service keeps the backend it already uses — μ/log for Kapital, Telemere for Digital Worker — by setting two vars at startup (*log-fn* and *context-fn*).

Log ids to preserve. Keep the structured ids the current code emits so existing dashboards/queries keep working where the same backend is used: :dbos.serializer/java-object-boxed, :workflow/step-start, :workflow/registration, :dbos/start, :dbos/stop. Namespacing them under :dbos.* in the library is the one intentional normalization.

5Consumer migration

5.1 Kapital (Mount)

Kapital's ohana.kapital.server.dbos.core defstate dbos becomes a thin wrapper: it computes the Kapital-specific inputs (executor-id, listen-queues, conductor, queue registry) and delegates construction/launch to the library. ohana.core.dbos.client's defstate wraps dbos.client/create-client. The workflow registry and dummy-workflow test namespace stay in Kapital.

(defstate dbos
  :start (let [inst (dbos.core/create
                     {:config   {:datasource kapital-ddl
                                 :app-name (:app-name cfg)
                                 :app-version (:app-version cfg)
                                 :executor-id dbos-executor-id
                                 :listen-queues (compute-listen-queues ...)
                                 :conductor (:conductor cfg)}
                      :queues   queue-registrations
                      :workflows workflow-registrations})]
           (dbos.core/launch! inst)
           (when (worker-executor? dbos-executor-id)
             (dbos.core/apply-schedules! inst workflow-registrations))
           inst)
  :stop  (dbos.core/shutdown! dbos))

5.2 Digital Worker (Integrant)

saas.dbos.core's ig/init-key :dbos/instance wraps create+launch!; ig/halt-key! wraps shutdown!. The #ig/refset :dbos/workflow collection is passed straight through as :workflows. No listen-queues/conductor/schedule — all optional.

(defmethod ig/init-key :dbos/instance
  [_ {:keys [db workflows app-name app-version schema admin-server?]}]
  (doto (dbos.core/create
         {:config {:datasource db :app-name app-name :app-version app-version
                   :executor-id executor-id :schema schema
                   :admin-server? admin-server?}
          :queues [(-> (Queue. global-queue-name) (.withWorkerConcurrency ...))]
          :workflows workflows})
    (dbos.core/launch!)))

(defmethod ig/halt-key! :dbos/instance [_ inst] (dbos.core/shutdown! inst))

Both keep their own queue names and workflow definitions. The lib never names a queue or defines a business workflow (only dummy ones for its own tests). Kapital's kapital-global-queue/kapital-dbos-worker-queue and Digital Worker's digital-worker-queue stay put.

7Decisions & risks

TopicDetail
DEC Stateless library No defstate/Integrant/config inside the lib — it exposes create/launch!/ shutdown!. Rationale: the duplication was in the interop, not the lifecycle; consumers already differ on lifecycle (Mount vs Integrant) and must keep owning it.
DEC Trove facade Library logs through trove/log!; consumers pick the backend via *log-fn*. Lets the two convenient step macros ship with logging while Kapital keeps μ/log and Digital Worker keeps Telemere. Trove already provides both backend fns.
DEC Queue names stay out Queue names and business workflow definitions are app policy, passed in by the consumer. The lib only ships dummy workflows for its own tests.
DEC Frozen wire format serializer-name = "transit_json_verbose" and the java-object box semantics are copied verbatim — existing rows in both services' system DBs must stay readable.
DEC No Malli — pure-Clojure validation The workflow-definition check is four keys; a hand-written predicate collecting error strings replaces Malli's mc/validate+me/humanize. Keeps the library's only hard deps to dev.dbos/transact, com.taoensso/trove, and Transit — no schema library forced on consumers. validate-workflow! keeps its log-then-throw contract.
DEC Protocol-based query list-workflows/get-workflow-status dispatch through a WorkflowQueryable protocol extended onto both DBOS and DBOSClient (both expose identical listWorkflows/getWorkflowStatus signatures). One API works on the in-process instance and the out-of-process client; open for test doubles. Replaces Kapital's client-only fns.
DEC De-complect events from progress Both repos named these helpers set-progress-event!/get-progress-event, baking a single use case (progress reporting) into the interface. DBOS events are a general key/value channel — progress is only one use of them. The lib exposes the primitive as set-event!/get-event; "progress" becomes a caller-side convention (a chosen event key), not part of the API surface.
DEC Injected *context-fn* Trove core has no with-context. Rather than lose Kapital's ambient step/workflow tagging, the lib exposes a dynamic *context-fn* [ctx thunk] (no-op default) that the step macros wrap their body in; consumers set it to u/with-context (μ/log) or t/with-ctx+ (Telemere) at startup. Backend-agnostic, one line to wire, preserves the nested-context behavior. See §4.1.
RISK Internal DBOS API drift The lib depends on DBOSIntegration (marked internal). A DBOS upgrade could change it. Mitigation: confine all such interop to dbos.core (already the case) and pin dev.dbos/transact explicitly; cover with the live integration test.
RISK Serializer behavior parity Subtle differences (fail-fast on atoms, Jackson round-trip probe, rethrow unwrapping) are easy to lose in a port. Mitigation: port both repos' serializer_test as the acceptance gate for T2 before any consumer cutover.

8Open questions

QuestionNotes
RESOLVED How much context propagation should the library offer? Resolved (§4.1): a pluggable *context-fn* hook — the lib owns the seam (step macros wrap their body in it) but not the mechanism; consumers plug in their backend's context macro. No-op by default, so it's zero-cost for callers who don't care.
RESOLVED ClojureScript surface? Resolved: none. dbos-transact is JVM-only, so the library is .clj-only. (Maven coordinates for the OSS release, e.g. io.github.ovistoica/dbos-clj, still TBD but non-blocking.)
RESOLVED Ship defworkflow + registry? (see §3.7) Resolved: no, not for now. The explicit definitions-vector path stays the only registration API. §3.7 is kept as recorded analysis in case a future consumer revisits it — but nothing ships, so dbos.core stays free of a load-time global.
RESOLVED Should the client have a lifecycle helper? Resolved: no. Only create-client is exposed. Lifecycle matters for the core DBOS instance — shutdown! must drain workflows still in flight — but a DBOSClient is a stateless enqueuer with no such obligation, so no launch/shutdown wrapper is warranted. Caller .closes it if/when they want.
RESOLVED Bundle logging backends? Resolved: no. The library depends on com.taoensso/trove only — never on a concrete logger. That is the entire point of Trove: a library stays backend-neutral and the consumer decides. Kapital adds μ/log, Digital Worker adds Telemere, each wires its own *log-fn*.
dbos-clj — extraction plan. Grounded in ohana.core.dbos.*, ohana.kapital.server.dbos.core, saas.dbos.*, and dev.dbos/transact 1.0.0 usage verified in-repo. Implementation may differ cosmetically.
